# 1단계: 데이터 수집 (Bronze Layer)

## 목표

KEPCO 공공 API에서 지역별 가구 평균 전력 사용량 데이터를 수집하여 **가공하지 않은 원본 그대로** MySQL에 저장합니다.

원본 데이터를 별도로 보관하는 이유는, 이후 가공 과정에서 문제가 발생했을 때 원인이 파이프라인의 버그인지, 아니면 원본 데이터 자체의 문제인지 구분하기 어렵기 때문입니다.

---

## 데이터 소스

- **API**: [가구평균 전력사용량 API](https://bigdata.kepco.co.kr/cmsmain.do?scode=S01&pcode=000493&pstate=house&redirect=Y)
- **엔드포인트**: `https://bigdata.kepco.co.kr/openapi/v1/powerUsage/houseAve.do`
- **수집 범위**: 2013년 5월 ~ 2025년 12월 (최초 데이터 제공 시점 기준)
- **지역 범위**: 전국 17개 광역시도 (시군구 단위)

### API 제공 필드

| 필드명 (API) | 저장 컬럼명 | 설명 |
|---|---|---|
| year | year | 연도 |
| month | month | 월 |
| metro | sd_name | 광역시도 |
| city | sgg_name | 시군구 |
| houseCnt | house_cnt | 해당 지역 가구 수 |
| powerUsage | power_usage | 가구 평균 전력 사용량 (kWh) |
| bill | bill | 가구 평균 전기 요금 (원) |

> API 응답에는 광역시도 코드(sd_code)가 포함되지 않아, 요청 시 사용한 지역 코드를 직접 추가하였습니다.

---

## MySQL 테이블 설계

```sql
CREATE TABLE household_power(
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    sd_code     TINYINT NOT NULL,
    year        SMALLINT NOT NULL,
    month       TINYINT NOT NULL,
    sd_name     VARCHAR(15) NOT NULL,
    sgg_name    VARCHAR(15) NOT NULL,
    house_cnt   INT NOT NULL,
    power_usage DECIMAL(10,2) NOT NULL,
    bill        MEDIUMINT NOT NULL
);

CREATE INDEX idx_time   ON household_power(year, month);
CREATE INDEX idx_region ON household_power(sd_name, sgg_name);
```

조회 시 자주 사용할 `year/month`와 `지역명` 기준으로 인덱스를 생성하였습니다.

## 수집 로직 (scripts/household_power_insert.ipynb)

### 핵심 처리 흐름

1. 17개 광역시도 코드 순회
2. **중복 수집 방지**: `is_already_collected(year, month, region_code)`로 이미 수집된 기간/지역은 건너뜀
3. API 요청 및 응답 처리
4. 응답 데이터에 지역 코드(`sd_code`) 추가
5. `convert_input()`으로 컬럼명 매핑 및 타입 변환
6. `save_batch()`로 MySQL에 배치 저장
7. API 요청 간 **5초 대기** (Rate Limit 준수)

### 오류 처리

- 요청 실패 시 `fail_list`에 기록
- 수집 중 중단되더라도 중복 체크 덕분에 이어서 재개 가능

---

## 수집 중 발견된 이상 데이터

수집 완료 후 대시보드로 시각화하던 중, **2015년 1월~2월** 데이터에서 전국 평균 대비 약 10배에 달하는 이상치를 발견하였습니다.

해당 기간의 원인을 확인하기 위해 뉴스를 검색하고 AI에게 해당 시기 특이사항을 문의하였으나, 관련 근거를 찾지 못하였습니다.

**결론**: 데이터 수집 자체가 잘못된 것으로 판단되며, 이 데이터는 이후 Transform 단계에서 결측치(NaN)로 처리합니다.
